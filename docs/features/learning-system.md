# Feature: Learning System

## Overview

CodeGate's learning system allows the review engine to adapt to project-specific patterns over time. It has two sides: **read** (suppress findings that match known dismissals) and **write** (auto-learn new dismissals from developer feedback). The system is file-based -- all state lives in the `.codereview/` directory in the repository.

## Directory Structure

```
.codereview/
  config.yml              # Learning system configuration
  conventions.md          # Team coding conventions (read by agent)
  dismissed.jsonl         # Dismissed finding patterns (one JSON object per line)
  learned-patterns.jsonl  # Auto-generated patterns from repeated dismissals
  modules/
    auth.yml              # Per-module config for src/auth/**
    api.yml               # Per-module config for src/api/**
```

## Dismissed Findings (`dismissed.jsonl`)

Each line in `dismissed.jsonl` is a JSON object representing a dismissed finding pattern. These are auto-generated by Phase 2 when fix verifications mark findings as `justified`.

```jsonl
{"dismissed_id": "d-a1b2c3d4", "file_pattern": "src/auth/**", "category": "security", "title_pattern": "broad exception handler", "reason": "Intentional top-level error handler", "dismissed_by": "auto:phase2", "dismissed_at": "2024-01-10T10:00:00Z", "dismiss_count": 3, "scope": "module"}
{"dismissed_id": "d-e5f6a7b8", "file_pattern": "src/utils/eval_sandbox.py", "category": "security", "title_pattern": "eval usage", "reason": "Sandboxed eval in controlled environment", "dismissed_by": "auto:phase2", "dismissed_at": "2024-01-12T14:30:00Z", "dismiss_count": 1, "scope": "file"}
```

### Fields

| Field | Required | Description |
|-------|----------|-------------|
| `dismissed_id` | yes | Stable ID: `d-` + SHA1 of `file:category:title` (first 8 hex chars). Generated by `_compute_dismissed_id()`. |
| `file_pattern` | yes | File path or glob pattern. Starts as exact path (`src/auth/login.py`), widens to directory glob (`src/auth/**`) on scope escalation. |
| `category` | yes | Finding category (e.g., `security`, `best_practices`) |
| `title_pattern` | yes | Finding title that was dismissed |
| `reason` | yes | Human-readable reason from the justified fix verification |
| `dismissed_by` | yes | `"auto:phase2"` for auto-detected dismissals |
| `dismissed_at` | yes | ISO 8601 timestamp of most recent dismissal |
| `dismiss_count` | yes | Number of times this pattern has been dismissed. Drives scope escalation. |
| `scope` | yes | `"file"` or `"module"` -- determines matching breadth |

### Matching Logic

During Phase 1, the agent checks each finding against dismissal rules. Matching uses:

1. **File pattern match** -- finding's file path is tested against the entry's `file_pattern` (exact path for file-scoped, glob for module-scoped)
2. **Category match** -- finding's category must equal the entry's `category`
3. **Title match** -- finding's title is compared to the entry's `title_pattern`

Matched findings are moved to `suppressed_findings[]` in findings.json with a `dismissed_id` pointing to the matching entry.

## Learned Patterns (`learned-patterns.jsonl`)

Auto-generated patterns from repeated dismissals. Format:

```jsonl
{"pattern_id": "lp-a1b2c3d4", "module": "src/auth/**", "category": "security", "description": "Auto-learned: 'broad exception' dismissed 3x", "source": "dismissed_3x", "evidence": ["d-abc12345"], "created_at": "2024-01-15T10:00:00Z", "confidence_modifier": -0.3}
```

Learned patterns are consumed by the agent during review -- they serve as additional context about known false positive patterns in the codebase. The `confidence_modifier` (range -1.0 to 0.0) reduces the confidence score of matching findings, making them more likely to be filtered out.

## Per-Module Config (`modules/*.yml`)

Each YAML file in `.codereview/modules/` configures review behavior for files matching a glob pattern:

```yaml
# .codereview/modules/auth.yml
glob: "src/auth/**"
severity_shift: +1          # Elevate all severities by 1 level
never_flag:
  - code_style              # Don't flag code style in auth module
always_flag:
  - security                # Always flag security regardless of confidence
focus_categories:
  - security
  - best_practices
min_confidence: 0.6         # Lower threshold for auth module
force_modes:
  - security                # Always activate security mode for auth files
max_per_file: 8             # Allow more findings per file in auth module
```

### Module Config Fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `glob` | string | required | File path glob pattern this module applies to |
| `severity_shift` | integer | 0 | Shift severity up (+) or down (-). +1 means suggestion->warning, warning->critical |
| `never_flag` | list[string] | [] | Categories to never flag in this module |
| `always_flag` | list[string] | [] | Categories to always include regardless of confidence |
| `focus_categories` | list[string] | [] | Categories to prioritize when cap is hit |
| `min_confidence` | float | 0.7 | Override minimum confidence threshold |
| `force_modes` | list[string] | [] | Review modes to always activate for this module |
| `max_per_file` | integer | 5 | Override per-file finding cap |

## Learning Write-Back

The write-back system is implemented in `post_findings.py` via several helper functions that form a pipeline.

### Step 1: Detect Dismissals (`_detect_dismissals`)

When `post_findings.py` processes fix verifications, it scans for `justified` entries:

1. Look up the original finding by `cr_id` in `findings_file.findings`
2. Compute a stable `dismissed_id` via `_compute_dismissed_id(file, category, title)` — produces `d-` + SHA1[:8]
3. Build a dismissal dict with `file_pattern` (exact file path), `category`, `title_pattern`, `reason`, `dismissed_by: "auto:phase2"`, `dismiss_count: 1`, `scope: "file"`

### Step 2: Merge Dismissals (`_merge_dismissals`)

New dismissals are merged with existing entries from `.cr/dismissed.jsonl`:

- **If `dismissed_id` exists:** increment `dismiss_count`, update `dismissed_at`
- **If new:** add to the index with `dismiss_count: 1`

### Step 3: Scope Escalation (within `_merge_dismissals`)

When `dismiss_count` reaches 3 for a file-scoped dismissal:
- `scope` is escalated from `"file"` to `"module"`
- `file_pattern` is broadened from exact path to parent directory glob via `_common_ancestor_glob()` (e.g., `src/auth/login.py` -> `src/auth/**`)

### Step 4: Auto-Pattern Generation (`_generate_learned_patterns`)

Groups dismissed entries by `(category, title_pattern)`. When total `dismiss_count` across the group reaches >= 3:

1. Compute `pattern_id` as `lp-` + SHA1 of `category:title_pattern` (first 8 hex chars)
2. Skip if pattern_id already exists in learned patterns
3. Compute `module` glob via `_common_ancestor_glob()` across all file_patterns in the group
4. Generate pattern with `confidence_modifier: -0.3`, `source: "dismissed_3x"`, and `evidence` list of dismissed_ids

### Step 5: Learning Comment (`_build_learning_comment_md` + `_post_learning_comment`)

Builds a structured PR comment with:
- New dismissed patterns (with JSONL blocks for copy-paste)
- Updated dismissed patterns (showing new dismiss_count)
- Scope escalations (entries that just crossed the module threshold)
- New learned patterns (with JSONL blocks)

The comment includes a `<!-- CODEGATE-learning-update -->` marker and instructions for applying the JSONL blocks to `.cr/dismissed.jsonl` and `.cr/learned-patterns.jsonl`.

### Implementation Status

Steps 1-4 execute in the `run()` flow — dismissals are detected, merged, and patterns are computed. However, the file write-back (persisting merged dismissals) and comment posting (Step 5) are implemented as helpers but **not yet wired** into `run()`. The computed results are available in local variables but not persisted or posted.

### Delivery Configuration

The `learning_delivery` field in `.codereview/config.yml` controls how learning artifacts are delivered:

| Value | Behavior |
|-------|----------|
| `comment` (default) | Learning changes are posted as a PR comment summarizing what was learned |
| `commit` | Learning changes are committed directly to the repo branch |
| `none` | Learning is disabled; no write-back occurs |

```yaml
# .codereview/config.yml
learning_delivery: comment
```

## `cr learn` CLI

The `cr learn` command provides manual management of the learning system:

### Commands

```bash
# List all current dismissals and learned patterns
cr learn --list

# Add a manual learned pattern
cr learn --add --pattern "broad exception handler" --category security --confidence-modifier -0.3

# Remove a dismissal or pattern by ID (d-xxx for dismissals, lp-xxx for learned patterns)
cr learn --remove d-abc12345
cr learn --remove lp-a1b2c3d4

# Analyze current dismissals: suggest patterns from repeated dismissals
cr learn --analyze

# Show learning system statistics
cr learn --stats
```

### `--list` Output

```
=== Dismissed findings (12) ===
  [d-abc12345] security / broad exception handler — src/auth/login.py:42
  [d-def67890] security / eval usage — src/utils/eval.py:15
  ...

=== Learned patterns (3) ===

  ...
```
